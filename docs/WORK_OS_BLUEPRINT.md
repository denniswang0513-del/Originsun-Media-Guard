## 1. 靈魂：專案生命週期狀態機（目前最大的結構缺口）

現在專案只有 4 狀態（洽談中/報價中/進行中/已結案），「進行中」把前期→拍攝→後期
全部黑箱。所有領域要串起來，靠的就是這條狀態主幹：

```
洽談 → 報價 → 成案 ──→ 前期籌備 → 拍攝 → 後期 → 內部審查 → 客戶審查 → 交付
 │      │      │           │         │      │                    │
 AM    報價簽核  自動開案:    企劃/通告  通告→   後期工具連動        │
 │    (✅已有)  成本子表+    /器材單    工時打卡  (產出回寫狀態)      ▼
 │              派工骨架     (§2)      (§3)                      結案
 │                                                            ┌───┴───┐
 └── 未成案 → 歸檔                                     官網上架收件匣  應收帳款
                                                        (✅已有)   → 收款完成
                                                                     │
                                                              獎金結算觸發 (§4)
```

**設計原則**：
- 狀態欄位擴充為兩層：`status`（商務層：洽談/報價/進行/結案/歸檔，**不動既有值**，
  相容全部現有查詢）+ 新增 `phase`（執行層：前期/拍攝/後期/審查/交付，僅「進行中」有值）。
- 每個狀態轉換有「誰能推」（見 §5 角色）與「推了觸發什麼」（自動化掛鉤清單見 §6）。
- 儀表板、工作台、獎金全部從這條主幹取數 — **先做這個，其他領域才有掛的地方**。

## 2.5 社群編輯自動化（第八領域，2026-07-07 追加）

**願景**：官網有新文章/新作品時，AI 每日自動規劃社群任務、產出多平台文稿，
人只做審核與發布。**完全複用既有 AI runner 骨架**（seo_runner / 翻譯的
「排程 claude 產出 → 後台審核卡 → 核准」模式，第三個應用）。

### 每日循環（social_runner，跑在 master、gate 同 seo_runner）

```
每日 HH:MM（排程）
  1. 掃內容源：新上架作品 / 新專欄文章 / 新公益案例（DB timestamp 比對水位）
  2. 無新內容 → 常青輪播：從「可宣傳內容庫」挑舊作品 re-share（頻率上限內）
  3. 每個選題 × 每個啟用平台 → claude 產文稿（平台語氣模板 + 品牌調性
     + 該內容的官網文案/SEO 描述當素材 + 建議配圖 URL）
  4. 寫入 social_posts（status=draft）→ Google Chat 每日摘要
     「今日社群任務：N 篇草稿待審」+ 公布欄指派官網編輯
  5. 編輯在後台審核卡：改稿 / 核准 / 退回重寫（同翻譯審核 UX）
  6. 核准 → 進發佈排程（黃金時段）→ 發佈器自動發文 → 回寫實際貼文連結
  7. 發佈失敗 → 告警推播 + 留佇列（可重試/切人工）
```

### 設計要點

- **自動發佈（2026-07-07 owner 拍板）**，風險用工程對策吃掉：
  - **通道**：優先走 **n8n 中繼**（NAS 既有容器）— 核准後 POST webhook 給
    n8n，由其 FB/IG/Threads 節點發文、回呼 published_url。平台 token 住在
    n8n 憑證庫，不進我們 DB；n8n workflow JSON 匯出存 repo（docs/n8n/）
    防「設定不在版控」。備選：Python 直連 Graph API（token 管理代碼量大）。
  - **核准 ≠ 立即發**：approved → 發佈排程（預設黃金時段 12:00/19:00，
    後台可調）→ 發佈器出手。給「核准後反悔」留時間窗。
  - **safety rails**：每日/平台上限在發佈器強制；`social.autopublish` kill
    switch 一鍵退回人工複製模式；發佈失敗告警（模板現成）；審核卡強制預覽。
  - **token 看門狗**：每日驗各平台 token 有效性，到期前 7 天推播提醒
    （Meta long-lived page token / IG / Threads 各有壽命）。
  - **平台可行性**：FB 粉專 + IG 商業帳號（Graph API，自家 app dev mode 對
    自有粉專可用；IG 需公開圖片 URL — 官網本身就能 host）+ Threads API ✅；
    **LinkedIn 公司頁要 Marketing Platform 審核，列後補**。IG 硬性要圖。
- **平台語氣分家**：IG（短+視覺+hashtag）/ FB（敘事長文）/ Threads（口語短打）
  各自模板，存 DB 可在後台編輯（同翻譯的品牌調性/術語表模式）。
- **客戶授權 gate**：只取「公開」作品（public projects 既有概念），另加
  `social_ok` 欄位 — 有些客戶案簽約不能宣傳，預設沿用公開狀態、可單獨關。
- **頻率政策**：每日/每平台上限（預設每平台每日 ≤1、每週 ≤4，後台可調），
  寧缺勿洗版；常青輪播同一作品 N 個月內不重複。
- **中英雙語**：官網已有英文版；社群 V1 先中文，英文帳號成立後掛翻譯 runner。

### DB 增量與檔案落點

  status[draft/approved/published/rejected], scheduled_date, published_url,

## 3. 工時紀錄（新領域一）

**定位決策（owner 先答，見 §7-D）**：內部成本分攤工具，**不是**勞基法出勤紀錄。
粒度：**天／半天**（影視現場切小時不現實，填報成本高會死）。

**採集策略 — 摩擦最小化是生死線**：
- 上游預填：排班日曆排了誰，當天工時就**預填好**（專案/日期/人/職務/整天），
  本人手機上「一鍵確認」或改成半天/加註。沒排班的臨時工時才手動新增。
- 內勤（剪輯/後期）：週視圖批次填（一週一次、按專案拖比例），不逼每日。
- 後期工具連動：機器任務歷史（job_history 已有）可輔助帶出「這台機今天在跑
  哪個專案」，供剪輯師填報參考（不自動計人時 — 機器時間≠人時）。

**流**：預填 → 本人確認 → 主管核可（週結）→ 鎖定 → 進專案成本（人日 × 日費，
日費取自人力庫，快照留存防日費調整污染歷史）。

DB 增量：`timesheets`（id, project_id, staff_id, work_date, portion[1.0/0.5],
role, day_rate_snapshot, status[draft/confirmed/approved/locked], approved_by,
source[schedule/manual]）。判定邏輯進 `core/crm_logic.py` 純函式 + 單元測試
（錢流慣例）。

### 3.5 工時 → 專案檢核迴圈（2026-07-08 追加 — 進行中就抓，不等結案）

工時資料不只餵成本，要**在專案進行中就回頭檢核專案**。三個數字對齊
（語意照 §7-F：報價=估、排班=排程、工時=實際）：

1. **專案工時檢核卡**（專案詳情新卡）：三條 burn bar — 報價人日（報價項目
   擴「職務×人日」欄）vs 排班人日 vs 已確認實際人日，並列消耗率與 phase 進度。
   「後期才做一半、工時已燒 90%」一眼看出。
2. **超支預警**（掛既有告警體系）：實際 ≥ 報價人日 80% 且 phase 未達後期
   → 提醒 PM；≥100% → 升級老闆。門檻後台可調。
3. **人力負載視圖**：週熱力圖 — 每人 × 專案的工時分佈，過載（>5 日/週）與
   閒置標色；派工缺口（下週要拍但無人 confirm）標紅。誰忙死誰閒著，不用問。
4. **填報異常檢核**：排班有、工時 48h 未確認 → 提醒本人；無排班的工時
   → 進 PM 待核清單（防漏填與幽靈工時，核可流本來就有，這裡做成可視清單）。
5. **餵 B2**：這些數字就是估價複盤的「即時版」，結案時快照留檔。

前置：報價項目需能標「職務×人日」（報價內部成本欄位小擴充，排進 N2 首段）；
其餘全部是 N2 資料的視圖層，**零額外填報負擔** — 檢核來自既有資料，不加人的工。

## 4. 獎金管理（新領域二）

**⚠️ 這裡程式是最後一步。獎金 = 政策，政策沒白紙黑字前寫任何 schema 都是猜**
（owner 作業清單見 §7-A）。系統設計上只承諾三件事：

1. **規則引擎**：政策定案後以純函式實作（`core/crm_logic.py` 模式 + 完整單元
   測試 — 這是薪酬，錯一筆傷信任）。輸入=專案結算數據（合約額/實收/外包/工時
   成本/雜支）+ 參與人員與角色，輸出=每人獎金建議數。
2. **結算單 + 簽核**：系統只產「建議單」，**發放永遠人核可**（owner 簽核 →
   進應付/請款流，複用既有帳務）。
3. **觸發點掛在生命週期**：預設「收款完成」觸發結算建議（不是結案 — 收款前
   發獎金等於公司墊錢，除非政策明說要墊）。

DB 增量：`bonus_rules`（版本化 — 政策會改，舊專案用舊規則）、`bonus_settlements`
+ items（每人明細、狀態[draft/approved/paid]、進請款單的關聯）。

## 7. 沒想周全的地方（按殺傷力排序）

**A. 獎金公式沒有定義 —— 這是 owner 作業，不是工程**。要白紙黑字回答：
基數是什麼（專案利潤%？固定額×職務？年度池？）、利潤怎麼算（含不含工時成本
/雜支/稅）、何時發（結案 vs 收款後）、分配給誰（只有派工的人？業務抽成？）、
虧損專案怎辦、中途換人怎麼分。**建議先用試算表跑三個真實已結案專案驗證公式**，
公式活了再進系統。

**B. 共用帳號 vs 個人化的正面衝突**。上週你決定五帳號全 admin 維持現狀 ——
但工時/獎金/「我的」工作台**必須知道操作者是誰**，共用帳號下整條路走不通。
需要回頭：每人一個 Google 帳號登入（OAuth 已就緒、成本極低），共用帳號降級為
公用電腦的唯讀/工具用途。這是整個藍圖的第一張骨牌。

**C. Row-level 權限是 RBAC 沒有的維度**。現在模組級全有全無；工時/獎金是
個人敏感資料：本人看自己、主管看團隊、財務看全部。要在守衛層加「scope」概念
（own/team/all），**設計一次共用**，別讓每個新模組自己發明。

**D. 工時的法律定位要先想清楚**。若它被當「出勤紀錄」，勞基法要求保存 5 年、
出勤到分鐘 — 系統責任完全不同。建議明確定位為「內部成本分攤」，出勤另有
既有機制（或不歸系統管），文件寫死避免日後被當出勤資料引用。

**E. 審核流目前不存在**。工時核可、獎金簽核、（未來）請款簽核 — 需要一個
極簡共用機制（狀態欄+核可人+時間戳，別做 BPM 引擎），一次設計三處復用。

**F. 「計畫 vs 實際」兩套數字會打架**。派工（計畫人日）、工時（實際人日）、
報價內部成本（估價）三者並存。定死語意：**報價用估、毛利與獎金用實際、
派工是排程工具**。儀表板永遠標明數字來源，否則財務會瘋。

**G. 通知個人化缺口**。Google Chat webhook 是廣播；「該你核工時了」需要到人。
第一版務實解：每人 user 檔存自己的 Chat DM webhook 或 @mention id，
notify 加收件人維度。公布欄同步長出「指派給個人」。

**H. 行動端不是選配**。現場人員（工時/通告/雜支）九成用手機 —— `/expense.html`
的獨立輕頁模式是對的，照抄；主 UI 的 RWD（Phase L）反而可以繼續等。

**I. 併發與衝突**。多人同時填工時/改排班 → 現有 inline-edit 是 last-write-wins。
工時表單人操作自己的列、天然低衝突；排班日曆要加樂觀鎖（updated_at 比對）。

**J. 器材/庫存的邊界**。前期的器材清單很容易滑坡成庫存管理系統（借還/保養/
折舊）。第一版就是 checklist，庫存明確列為「不做」，除非之後獨立立項。

**K. 平台底座債務跟業務擴張賽跑**。ROADMAP 既有的佇列持久化（master 重啟丟
任務）、master 去單點化、機隊推版 — 系統越裝越多公司命脈，這些底座項的
優先級只會越來越高，別讓功能一直插隊。

**L. 上線策略：影子運行**。工時/獎金第一個月與現行做法（口頭/試算表）並跑，
月底對帳一致才切正式 — 薪酬相關系統不能 big bang。

## 10. 做完後的樣子（終局圖景 — 全部做完，公司變成這樣）

### 一個案子的完整旅程（全程在系統裡，沒有斷點）

1. **案子自己走進來**：產業情報 runner 早上 08:30 掃到一個文化部標案，
   評分 87、截止日倒數 12 天，紅字掛在「📡 產業情報」頂端。業務按「轉提案」，
   提案庫開了草稿，AI 已把過往兩個同類得標提案和參考片單掛在旁邊。
2. **提案 → 成案零重輸**：提了案、中了。按「成案」，CRM 專案自動建好——
   客戶、報價、付款節點（訂金 30% 已標「待請」）、成本子表、派工骨架全部生成。
3. **前期不再靠記憶**：製片開場景庫搜「工廠 桃園」，三個勘過的場地帶著照片、
   費用、上次的踩雷紀錄跳出來。排班日曆選人時系統標紅了檔期衝突，
   通告單自動生成，外部攝影師手機收到 token 連結。
4. **拍攝日自動記錄**：現場人員手機工作台：今日通告、一鍵確認工時（半天/整天）、
   雜支拍照上傳。器材出庫有紀錄，晚上歸還自動核銷。
5. **素材自己排隊**：卡一插，備份 + 轉檔 + 逐字稿自動跑完，後期工作台出現
   「素材就緒」待辦。剪輯師要補一顆海岸空拍——素材庫搜兩個字，三年前的片段
   帶著縮圖條和時間點出現。
6. **審批不再猜**：初剪好了，客戶收到審批門戶連結，在 1:23 標「logo 再大一點」。
   全部意見 resolved、客戶按核准，專案 phase 自動推進，系統提醒「該請尾款了」。
7. **結案即行銷**：按結案——官網編輯收到收件匣通知，AI 已寫好 showcase 草稿
   和英文版；上架後系統自己驗證對外頁回 200；隔天早上社群 runner 產好
   FB/IG/Threads 三平台文稿，編輯改兩個字按核准，中午貼文自動出現在粉專。
8. **錢自己對帳**：尾款入帳，獎金結算建議單自動生成——按當時的費率快照算，
   owner 簽核後進請款流。月底 F1 月結鎖帳，複盤卡顯示「此案報價 vs 實際 +6%」。

### 每個人登入看到的（不再是 14 個 tab）

- **老闆**：一頁儀表板——專案管線泳道、90 天現金流、本月損益、待簽核、
  前三大客戶占比警示。決策要的數字不用問人。
- **業務**：我的客戶動態、報價追蹤、產業情報高分項、提案轉換率。
- **製片**：進行中專案看板（一眼看出卡在哪）、今日通告、待核工時。
- **現場人員**（手機）：今日通告 + 一鍵工時 + 雜支拍照，三件事一頁完。
- **剪輯**：待處理素材佇列 + 8 工具 + 素材庫檢索。
- **財務**：應收帳齡、待入帳、獎金結算單、月結按鈕。
- **官網編輯**：上架收件匣 + 社群審核佇列 + SEO 待審。

### 系統自己在做的事（五個 runner + 掛鉤，人只審核）

AI SEO、英文翻譯、社群文稿、產業情報、客戶關懷五個 runner 每天各自巡邏；
九個生命週期掛鉤讓「上一步做完，下一步已經生好在等你」。告警有人的名字，
提醒到個人。每季系統自動打包發票憑證給會計師。

### 公司層面的改變（可量測）

| 指標 | 之前 | 之後 |
|------|------|------|
| 結案→官網上架時滯 | 數週～數月 | < 48 小時（自動驗證） |
| 忘記請款 | 靠記憶 | 零（節點+提醒） |
| 專案毛利 | 虛高（無工時/折舊） | 真實（費率快照+攤提） |
| 估價準確度 | 憑感覺 | 每案複盤、逐季收斂 |
| 案源 | 被動等詢問 | 標案/補助自動進 pipeline + 舊客戶自動回訪 |
| 獎金 | 黑箱 | 規則透明、可稽核、自動建議 |
| 老闆的資訊成本 | 問人、翻試算表 | 登入即見，一頁 |

**一句話**：現在是「一套工具 + 一群靠記憶運轉的人」；做完後是
「一套會自己往前推的作業系統 + 專注在創作和決策的人」。
每個環節的資料都在累積成資產——場景、提案、素材、費率、複盤——
公司規模翻倍時，這套系統不用重蓋，加人加帳號就好。
